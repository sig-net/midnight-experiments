# Analysis — what actually drives Compact circuit cost

Interpretation of [REPORT.md](REPORT.md) (run `2026-07-14T05:30:45.884Z`,
compactc 0.33 / midnight-js 5.0.0-beta.3, local proof server on an M-series
Mac). REPORT.md is regenerated by `yarn report`; this file is hand-written.

## The headline findings

### 1. Proof SIZE is constant — stop worrying about it

Every single-call transaction produced a **2,940-byte proof**, whether the
circuit was the 15-instruction `noop` or the 943-instruction `emit4`
(persistent-hash circuits: 4,508 B, still constant across 32→1024 B inputs).
Proof size scales **only with the number of contract calls in the
transaction** (one proof per call: `callTwice` = 3 proofs = 8,820 B), never
with circuit complexity. Verifier keys (the on-chain footprint) are similarly
flat: 1,351 B / 2,119 B.

**Design consequence:** proof size is a non-factor when choosing between
Compact constructs. Proving TIME and prover-key size are what vary — by
orders of magnitude.

### 2. Proving time ≈ linear in prover-key size, with power-of-2 tiers

Prove wall-time tracks the prover key (≈ padded circuit rows) at roughly
**0.2 s per MB** on this machine. Rows pad to the next power of two, so cost
moves in **tiers**: `emit2` and `emit4` land in the same 67 MB tier and prove
in the same ~12–14 s — the 3rd and 4th events were effectively free once the
2nd crossed the tier boundary. Check your tier before micro-optimizing.

### 3. Events are the most expensive thing you can do in a circuit

The single biggest result. Adding ONE `Misc` event to the base workload:

| | base | +1 event | +2 events | +4 events |
|---|---|---|---|---|
| prover key | 147 KB | **33.7 MB (230×)** | 67.4 MB | 67.5 MB |
| prove time | 0.12 s | **6.7 s (57×)** | 14.5 s | 12.4 s |

The `Misc` payload is **fixed at `Bytes<256>`** (probed: `serialize<T, 32>`
into `payload` is a type error) — you pay the full 256-byte event cost even
to emit 24 meaningful bytes, and "bigger events" only exist as *more* events,
each adding ~33 MB of pre-padding rows. The cost is NOT the `serialize` (that
is trivial) — it is the event/log transcript commitment hashing baked into
`emit`.

**Design consequence:** treat `emit` as a ~7 s / 33 MB line item. Emit one
event, not several; never emit redundantly (e.g. don't emit what an
authenticated ledger read can serve). For the signet flow: an event as a
*notification ping* is affordable once per request, but emitting per-part or
per-chunk event streams (the `emitPart` pattern, N×256 B) multiplies proving
time per 256-byte step until a tier boundary absorbs it.

### 4. Cross-contract calls are cheap — 40× cheaper than an event

Adding one cross-contract call to the base workload: +26 zkir instructions,
prover key 147 KB → 280 KB, prove **0.12 s → 0.27 s** (caller proof + callee
proof, ~0.13 s each). A second call doubles it. The per-call overhead on the
caller side (argument marshalling + communication commitment) is tiny; the
real cost is simply that **the callee's own circuit must also prove** — the
tx carries one proof (+2,940 B) per contract in the tree.

`callEmit` shows the composition perfectly: the caller's proof stays at
0.18 s while the callee's `depositEmit` proof costs 5.17 s — **the event
dominates; the call is noise**.

**Design consequence for vault → signet routing:** the cross-contract call
mechanism itself adds ~0.15 s prove + 3 KB tx size. Centralizing request
emission in a signet contract is performance-cheap; what matters is how many
*events* the callee emits, not that it is called cross-contract.

### 5. Hashing: `transientHash` is free, `persistentHash` is not

Against same-size no-hash controls:

- `transientHash` (Poseidon, `Field` out) — **statistically free** at every
  size (0.92–1.02× the control). Use it for any in-circuit commitment that
  doesn't need a byte-exact persistent digest.
- `persistentHash` (SHA-256-class, `Bytes<32>` out) — 5× prove at 32 B
  (+0.5 s), and cost grows with input: 1024 B costs 3.95 s (19.4 MB key).
  Also the only construct that changed proof/verifier-key size (4,508 B /
  2,119 B — one extra lookup/table column).
- Shape doesn't matter: hashing `Vector<8, Bytes<32>>` ≡ flat `Bytes<256>`.

### 6. Large circuit INPUTS are their own cost — even unused

The controls take `Bytes<N>` and ignore it: 32 B → 147 KB key / 0.12 s;
256 B → 1.07 MB / 0.29 s; 1024 B → 4.2 MB / 0.79 s. Every input byte is
constrained whether or not you use it. The same holds across a contract
boundary (`callBig`, 256 B arg: both caller and callee pay ~0.37 s each).

**Design consequence:** keep circuit arguments (and cross-contract
arguments) as narrow as possible; pass commitments/digests instead of blobs
when the data doesn't need to be constrained in-circuit.

### 7. zkir instruction count is a MISLEADING size metric

`persistent1024` is 69 zkir instructions but a 19.4 MB prover key;
`emit4` is 943 instructions and 67 MB. Builtins (hashes, event commitments)
expand inside *constraints*, invisible to instruction count. **Use prover-key
bytes as the static size proxy**, instruction counts only to attribute which
statement grew.

### 8. End-to-end latency is dominated by the chain, not the prover

Every callTx costs ~18 s baseline (block cadence + indexer + wallet
balancing) regardless of circuit. Prove time only becomes user-visible once
it exceeds ~5 s — which today means: events, and kilobyte-scale
`persistentHash`. Everything else disappears into chain latency.

## Cost cheat-sheet (prove-time deltas over base workload, this machine)

| Construct | Δ prove | Δ prover key | Δ proof bytes |
|---|---|---|---|
| base workload itself (vs noop) | +0.0–0.1 s | +133 KB | 0 |
| 1 cross-contract call | +0.15 s | +133 KB (+ callee's own circuit) | +2,940 B |
| `transientHash`, any size ≤1 KB | ~0 | ~0 | 0 |
| `persistentHash<Bytes<32>>` | +0.5 s | +2.7 MB | +1,568 B (once) |
| `persistentHash<Bytes<1024>>` | +3.2 s | +15 MB | +1,568 B (once) |
| 256 B circuit/call argument | +0.2–0.6 s | +0.9 MB | 0 |
| **1 `Misc` event** | **+6.6 s** | **+33.6 MB** | **0** |
| 2nd `Misc` event | +7.8 s | +33.7 MB | 0 |
| 3rd & 4th events (same tier) | ~0 | ~0 | 0 |

## Caveats

- reps = 2 per circuit; prove times have ±20% run-to-run noise (visible as
  emit2 "slower" than emit4 — same tier, same true cost). Tier boundaries and
  orders of magnitude are robust; second-decimal comparisons are not.
- Local proof server on one machine; absolute seconds scale with hardware,
  ratios should transfer.
- ledger-9 RC toolchain (see version matrix in REPORT.md); re-run on bumps.
